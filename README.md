Tratic
=====================================

차트 기반 매매일지 분석 앱

### 배경

매매 일지를 작성하는게 좋다는 것은 누구나 아는데 왜 작성하지 않을까?  
매매 일지를 작성하는건 귀찮고, 매매 일지 작성의 효과가 눈에 드러나지 않기 때문에  

### 앱의 목표

기술 지표 분석을 통한 매매 기록의 즉각적, 정량적 평가  

### 흐름

1. 유저 로그인
2. 매매기록 입력
3. 차트 호출
4. 지표 생성
5. 매매기록 분석

### 구현

차트를 일일이 저장하지 않고, 요청이 들어오면 해당 시점 기준으로 과거 캔들을 분석에 필요한 만큼 조회한다.
초기 입력 모델은 손절가와 포지션 비중까지 받는 방향을 검토했지만, 현재는 진입 자체의 방향성과 차트 문맥 판단에 집중하기 위해 입력을 단순화하는 쪽으로 정리하고 있다.

#### 사용자에게 받는 입력
- 마켓
- 심볼
- 시간
- 사용한 시간봉
- 진입가
- 방향(Long/Short)

입력 단순화를 우선하며, 손절가와 포지션 비중은 현재 기본 입력에서 제외하는 방향으로 정리 중이다.
심볼 입력은 시장별 포맷 차이(예: `KRW-BTC`, `BTCUSDT`)가 크기 때문에, 추후 심볼 검색/자동완성은 마켓 선택과 결합된 형태로 제공하는 것을 전제로 한다.

#### 사용자가 받는 출력
리포트 형식의 거래 판단

현재 결과 화면은 최종 시나리오와 요약 문장 중심으로 유지하되, 이후에는 raw 점수 자체보다 해석 가능한 4축 라벨링 결과를 함께 보여주는 방향으로 정리한다.
- 추세(trend)
- 변동성(volatility)
- 위치(location)
- 수급 압력(pressure)

#### 보장하는 해상도
일봉까지. 단, 일봉 아래 존재하는 해상도는 사용자가 직접 확인해야함

#### 레이트리밋
마켓 API 레이트 리밋 또는 연속된 내부 서버 에러 발생 시 요청을 일정시간 잠굴 것

### 디렉토리 구조

- `frontend/`: React + Vite 프론트엔드
- `backend/`: Spring Boot 백엔드
- `backend/build/generated/frontend-resources/main/static`: 프론트 빌드 산출물이 합쳐지는 생성 디렉토리

### 로컬 실행

1. `fnm install --lts`
2. `cd frontend`
3. `fnm exec --using=.node-version npm ci`
4. 프론트 개발 서버: `fnm exec --using=.node-version npm run dev`
5. 백엔드: `cd ../backend && ./gradlew bootRun`

### 패키징

- 전체 자동 빌드: `cd backend && ./gradlew build`
- 프론트만 갱신: `cd backend && ./gradlew syncFrontendAssets`
- React 산출물은 빌드 시 Spring Boot 정적 리소스로 포함된다.
