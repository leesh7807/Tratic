# Frontend Concept Guide

## 목적
- 이 문서는 현재 프론트 작업에서 유지해야 할 제품 컨셉과 화면 철학만 정리한다.
- 시각 스타일, 컴포넌트 디자인, 세부 레이아웃은 의도적으로 다루지 않는다.
- 이후 디자인을 전면 교체하더라도, 아래 개념적 기준은 쉽게 흔들리지 않는 것을 목표로 한다.

## 제품 정체성
- Tratic 프론트는 실시간 매매 실행 도구가 아니라, 거래 기록을 입력하면 그 setup에 대한 해석 결과를 바로 확인하고 되돌아보게 하는 분석 저널이다.
- 첫 세션에서 사용자가 받아야 하는 핵심 가치는 `즉각적 해석`이다.
- 제품은 사용자를 평가하거나 선고하는 도구처럼 보이지 않고, 기록과 복기를 돕는 도구처럼 보이는 쪽을 택한다.

## 사용자 흐름
- 기본 흐름은 `거래 기록 입력 -> 해석 결과 확인`이다.
- 비로그인 사용자도 이 기본 흐름을 끝까지 사용할 수 있어야 한다.
- 로그인은 사용 가능 여부를 가르는 문턱이 아니라, 결과 저장과 기록 누적 가치를 추가하는 단계로 둔다.
- 첫 화면은 사용자가 바로 입력을 시작할 수 있게 해야 하며, 결과는 같은 흐름 안에서 이어져야 한다.

## 입력 철학
- 입력 계약은 메인 브랜치의 최신 `README.md`와 백엔드 도메인 변경을 우선 따라간다.
- 현재 재구축 기준의 기본 입력은 `market`, `symbol`, `entryAt`, `entryPrice`, `direction` 중심의 단순 입력을 우선 검토한다.
- `resolution`은 메인 표의 기본 컬럼이 아닐 수 있지만, 입력/분석 흐름 어디에 둘지는 별도 UI 결정으로 남긴다.
- 입력 경험의 핵심은 “많은 옵션 제공”이 아니라 “헷갈리지 않는 기본 흐름”이다.
- 심볼 입력은 검색형 선택으로 확장 가능해야 하며, 이는 이후 API와 데이터 연결이 붙더라도 유지할 전제다.

## 결과 철학
- 결과의 핵심은 `scenario`와 `summary`다.
- `scenario`는 저장, 통계, 재조회에 쓰이는 안정 식별자다.
- `summary`는 사용자에게 보여주는 설명 문구이며 교체 가능한 표현값이다.
- 프론트는 내부 메타데이터(`bias`, `confidence`, `riskLevel`)보다 외부 결과의 명확한 해석 전달을 우선한다.
- raw 지표값 나열보다 “이 setup을 어떻게 읽어야 하는가”를 먼저 전달하는 쪽을 택한다.
- 다만 메인 브랜치 기준으로 `trend`, `volatility`, `location`, `pressure` 4축 라벨링이 함께 노출될 수 있음을 전제로 결과 영역을 분리 가능하게 설계한다.

## 표현 원칙
- 결과는 사용자가 한눈에 읽을 수 있어야 하며, 표현은 언제나 `scenario`를 보조하는 방향으로만 사용한다.
- 어떤 시각 요소가 들어가더라도 그것이 `scenario`를 대체해서는 안 된다.
- 결과 표현은 안정 식별자와 사용자 대상 문구를 분리해서 다뤄야 한다.
- 저장, 분기, 통계 기준은 항상 `scenario` 원본을 사용하고, 번역명이나 장식적 표현은 기준값으로 사용하지 않는다.

## 톤과 카피
- 제품 카피는 단정적 판단보다 가능성과 해석을 전달하는 쪽을 택한다.
- `맞았다`, `틀렸다`, `절대적` 같은 표현은 피한다.
- 학습 효과가 생길 수는 있지만, 제품이 학습 서비스처럼 보이게 만들지는 않는다.
- 기록과 복기를 돕는 어조를 기본으로 유지한다.

## 계약과 확장
- 생성된 OpenAPI JSON은 참고자료이며, 실제 계약 판단은 백엔드 도메인/컨트롤러와 현재 구현을 우선 확인한다.
- 입력 계약과 결과 모델이 바뀌는 과도기에는 UI가 이전 목업보다 최신 도메인 문서를 먼저 따른다.
- 내부 시나리오 식별자는 영어를 유지하고, 프론트에서는 필요하면 별도 번역 레이어를 통해 한국어로 표현한다.
- 디자인이 바뀌더라도 `입력과 결과를 한 흐름으로 묶는 것`, `scenario 중심으로 읽히게 하는 것`, `비로그인도 분석 가능하게 두는 것`은 쉽게 바꾸지 않는다.

## 문서 유지 원칙
- 이 문서는 코드나 스크린샷 없이도 설명 가능한 컨셉만 남긴다.
- 디자인 디테일, spacing, 색상, 버튼 모양, 패널 위치 같은 내용은 이 문서에 남기지 않는다.
- 제품 컨셉이 바뀌면 이 문서를 먼저 갱신하고, 스타일 조정만 있을 때는 이 문서를 건드리지 않는다.
